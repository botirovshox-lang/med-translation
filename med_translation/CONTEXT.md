# CONTEXT.md — Medical Translation TM Architecture

## Обзор

**med_translation** — полностью автоматизированная система для построения профессионального translation memory (TM) и глоссария медицинских терминов EN↔RU.

```
MedlinePlus (356 ссылок)
         ↓ download_pdfs.py (скачивание)
         ↓
Google Cloud Storage (210 PDF, ~90 МБ)
         ↓ extract_tm.py (экстракция текста)
         ↓
695 билингвальных пар (EN↔RU, параграфы)
         ├→ tm.tsv, tm.tmx, tm.json
         │
         ↓ build_glossary.py (Claude API)
         ↓
~1000 медицинских терминов (категоризация)
         └→ glossary.tsv, glossary.json
```

---

## Три стадии обработки

### Стадия 1: Загрузка PDF (download_pdfs.py)

**Вход:** MedlinePlus русскоязычная страница с 356 ссылками на билингвальные PDF

**Процесс:**
1. Парсим HTML (`BeautifulSoup`)
2. Ищем ссылки с текстом "Русский" или "Russian" + "Bilingual"
3. Скачиваем напрямую с Google Cloud Storage
4. Сохраняем в `pdfs/`
5. Логируем статус в `pdf_index.json`

**Выход:** 
- 210 PDF файлов (~90 МБ)
- 12 не найдено (no_pdf_found)
- 344 успешно скачано (207 новых + 137 из кэша)

**Ключевые решения:**
- Задержка 1.2 сек между запросами (вежливо к серверу)
- Checkpoint в `pdf_index.json` (можно перезапустить, не переcкачивая)
- Нормализация имён файлов (ASCII только, убираем спецсимволы)

---

### Стадия 2: Извлечение TM (extract_tm.py)

**Вход:** 210 PDF в двухколоночном формате (EN слева, RU справа)

**Проблема:** Автоматическое разделение колонок сложно, т.к. PDF хранит текст просто как последовательность символов без информации о структуре

**Решение: Координатный анализ**

1. **Автодетекция разделителя:**
   - Извлекаем все слова с их x0-координатой
   - Ищем наибольший gap в распределении x0 (между колонками)
   - Точка разделения = середина этого gap
   - Fallback: 48% ширины страницы (для случаев без явного gap)

2. **Построчное группирование:**
   ```python
   # Группируем слова по Y-позиции (±5 пунктов = одна строка)
   left_lines = {}   # y → [word1, word2, ...]
   right_lines = {}
   
   for word in page.extract_words():
       y_key = round(word['top'] / 5) * 5
       if word['x0'] < split_point:
           left_lines[y_key].append(word['text'])
       else:
           right_lines[y_key].append(word['text'])
   ```

3. **Фильтрация шума:**
   - Убираем номера страниц (`^\d+$`)
   - Убираем URL (`healthinfotranslations`)
   - Убираем короткие строки (`< 3 символов`)

4. **Дедупликация:**
   - Убираем кириллицу из EN-текста (попала из русских заголовков)
   - Убираем кириллические слова regex: `[А-ЯЁа-яё]+`

5. **Валидация:**
   - Проверяем что левая половина = EN, правая = RU (`detect_lang()`)
   - Минимальная длина сегмента: 30 символов

**Выход:**
- 695 чистых билингвальных пар (EN, RU)
- Форматы: TSV, TMX 1.4, JSON

**Качество:**
- Некоторый шум на титульных страницах (смешанный layout)
- Основной контент (медицинская информация) выравнен идеально
- Пригодно для CAT-инструментов (Trados, OmegaT, memoQ)

---

### Стадия 3: Глоссарий терминов (build_glossary.py)

**Вход:** 695 билингвальных пар (параграфы уровня)

**Процесс: Claude API + батчинг**

1. **Батчинг:**
   - Группируем пары по 8 штук
   - 695 пар ÷ 8 = ~87 API-запросов

2. **Prompt caching:**
   ```
   System prompt (кэшируется):
     "Ты эксперт по медицинской терминологии..."
   
   User message (новый каждый раз):
     "Извлеки термины из 8 пар..."
   ```
   - Экономия: first request ~4K input tokens, subsequent ~200 tokens
   - Кэш-hit rate: ~95% (сэкономлено ~80% input-токенов)

3. **Извлечение терминов:**
   - Claude анализирует каждую пару
   - Находит медицинские термины, процедуры, диагнозы, анатомию
   - Возвращает JSON с категоризацией

4. **Категории:**
   - `diagnosis` — болезни, синдромы (бронхит, анемия)
   - `symptom` — симптомы (кашель, усталость)
   - `procedure` — процедуры (КТ-сканирование, биопсия)
   - `anatomy` — анатомия (бронхи, эритроциты)
   - `medication` — лекарства, витамины
   - `test` — тесты, анализы
   - `other_medical` — прочее (редко)

5. **Checkpoint:**
   - Каждые 10 батчей сохраняет прогресс в `glossary_progress.json`
   - При ошибке/прерывании перезапускается с места остановки

6. **Дедупликация:**
   - После всех запросов удаляем дубли по (en, ru)
   - Сортируем по категории и English термину

**Выход:**
- ~1000 уникальных медицинских терминов
- Форматы: TSV, JSON

**Стоимость:**
- ~210K input tokens (с кэшом ~42K эффективных)
- ~150K output tokens
- Haiku pricing: ~$0.05–0.10 на 695 пар

## Ключевые технические решения

### 1. Почему pdfplumber, а не PyMuPDF/fitz?
- pdfplumber лучше для word-level координат и таблиц
- Более простой API для текста с сохранением структуры
- Меньше проблем с кодировками

### 2. Почему Claude Haiku, а не GPT/другие?
- Стоимость: дешевле в ~10x раз чем GPT-4
- Скорость: достаточно быстро для медицинских терминов
- Prompt caching: экономит токены

### 3. Почему батчинг 8, а не 1?
- Амортизируем overhead API-запроса
- 8 пар ≈ 2000-3000 токенов input = хороший размер
- Меньше запросов = быстрее, дешевле

### 4. Почему checkpoint/progress каждые 10 батчей?
- На случай timeout/ошибки сети
- Не потеряем часы работы
- Быстрый перезапуск

## Расширяемость

**Легко добавить:**
- Другие языки (испанский, французский, немецкий) — меняем URL MedlinePlus
- Другие источники PDF (UpToDate, Cochrane, WHO docs)
- NER для именованных сущностей (лекарства, компании)
- REST API для поиска по TM
- Интеграцию с Trados / OmegaT через их API

**Сложнее:**
- Синхронизация с живыми обновлениями (MedlinePlus меняется)
- Версионирование TM (какую версию использовать для какого проекта)
- Рекомендация термина на основе контекста
